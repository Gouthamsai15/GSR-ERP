GSR ERP MASTER PROJECT PROMPT

Use this document when handing the codebase to an AI assistant, a new developer, or a future implementation agent.

The goal of this file is not to be short. The goal is to make the project legible end-to-end:
- what the product is
- how the frontend is structured
- how roles and routes work
- which page does what
- how the theme system works
- how Firebase, Firestore, the Node admin API, and legacy Supabase-shaped code interact
- which files are the primary source of truth

======================================================================
1. PROJECT IDENTITY
======================================================================

Project name:
- GSR ERP

Project category:
- Multi-role school ERP
- Multi-school SaaS-style education platform

Primary product purpose:
- Give schools one unified system for admissions, student records, staff records, classes, attendance, timetable, exams, results, fees, transport, notifications, audit logs, billing, and school subscription operations.

Core product audiences:
- Public visitors on the marketing home page
- Super Admin platform owner
- School Admin
- Principal
- Staff
- Student
- Parent

Product posture:
- This is intended to feel like a real operational institution platform, not a toy CRUD dashboard.
- The codebase is already large and should be treated as a working product foundation.
- The repository is in an active migration phase: Firebase is now the primary direction, but some service shapes and backend fallbacks still reflect the earlier Supabase structure.

Current product style:
- Public home page: bright, soft green, polished, mobile-tuned marketing page
- Login page: calm school-branded auth shell with green educational tone
- School dashboard: light sage / green “Fresh Meadow” theme
- Super Admin console: white + orange + navy “Solar Command” theme
- Legacy landing/theming scaffolding still exists in CSS, but the live home page is now its own brighter scoped design

4. APP ENTRY AND BOOT FLOW
======================================================================

Primary boot file:
- `src/main.tsx`

Boot behavior:
- mounts `App` inside `BrowserRouter`
- wraps the app with `LanguageProvider`
- imports the global stylesheet `src/styles/app.css`
- marks the document as ready with `document.documentElement.dataset.appReady = "true"`
- removes the boot placeholder element `#app-boot`

Application root:
- `src/App.tsx`

What `App.tsx` is responsible for:
- route tree
- auth hydration
- Firebase project sync on startup
- restoring session
- listening to Firebase auth state changes
- redirecting users by role
- protecting routes

Auth/session initialization in `App.tsx`:
- calls `syncFirebaseProjectConnection()`
- attempts `restoreSession()`
- subscribes to auth state via `subscribeToAuthSessionChanges(...)`
- hydrates role/school/session data through `hydrateAuthSession(...)`
- writes auth state into `authStore`

Public route guard:
- `PublicRoute` redirects authenticated users away from `/login`

Protected route layers:
- `ProtectedRoute`
- `RoleProtectedRoute`

12. AUTHENTICATION ARCHITECTURE
======================================================================

Primary auth file:
- `src/services/authService.ts`

Auth direction:
- Firebase Auth is the live auth backbone

What auth service does:
- login with identifier
- restore session
- hydrate role/user/school context
- subscribe to Firebase auth state changes
- support bootstrap super admin
- resolve school profile
- hit admin API when needed

Important auth facts:
- no public signup
- login can be email-based or student-ID based
- `AuthUser` is the normalized frontend identity shape
- `CurrentSchool` is resolved and attached to the store
- session includes Firebase token data

Admin API usage in auth:
- auth service can call the admin API to get authoritative user/school context
- `getAdminApiEndpoints()` and related helpers define endpoint selection

Firebase project hygiene:
- `syncFirebaseProjectConnection()` clears stale Firebase persistence when the configured Firebase project changes

Persistence:
- `ensureFirebasePersistence()` uses browser local persistence

13. DATA LAYER ARCHITECTURE
======================================================================

This repository has a hybrid data layer.

Primary modern direction:
- Firebase client auth + Firestore on frontend
- Firebase Admin on server

Legacy / migration support:
- Supabase client still exists
- SQL schema still exists
- some service flows still support SQL-shaped query patterns

Important files:
- `src/services/firebaseClient.ts`
- `src/services/firebaseService.ts`
- `src/services/adminService.ts`
- `src/services/databaseClient.ts`
- `src/services/saasService.ts`
- `server/firebaseAdmin.mjs`
- `server/adminApi.mjs`

Client database client:
- `databaseClient.ts`
- creates Supabase client if `VITE_DATABASE_URL` and `VITE_DATABASE_ANON_KEY` are present
- adds fetch timeout protection

Firebase client:
- `firebaseClient.ts`
- initializes:
  - Firebase app
  - Firebase Auth
  - Firestore
  - analytics on supported platforms

Operational service layer:
- `adminService.ts`
- this is the biggest and most important business service file in the repository

What `adminService.ts` covers:
- classes
- subjects
- holidays
- staff
- students
- attendance
- exams
- results
- fees
- employees
- staff attendance
- leaves
- audit logs
- notifications
- salary
- vehicles
- routes
- applicants
- dashboard preload
- timetable
- timetable impacts
- helper caches
- school-scoped reads
- Firebase and server-action fallbacks

Service pattern used across the file:
- if Firebase is available, read/write Firestore directly or through cached helpers
- if Firestore permissions block an operation, server API fallback may be used
- if Firebase is not available or feature is legacy-shaped, SQL/Supabase path may still be used

Important architectural consequence:
- developers must not assume one single backend path for all modules
- each service function may be:
  - Firebase-only
  - Firebase-first with server fallback
  - SQL-backed
  - migration-aware hybrid

22. IMPLEMENTATION GUIDELINES FOR FUTURE WORK
======================================================================

When modifying this repo, follow these rules:

1. Read before editing
- Always inspect the relevant page component, the related service, and `app.css` before making assumptions.

2. Respect theme scoping
- Dashboard school pages should stay in the school theme.
- Super Admin pages should stay in the super admin theme.
- Auth pages should stay in auth theme.
- Home page should use the bright scoped landing-home system.

3. Prefer CSS centralization
- Visual behavior should usually go into `src/styles/app.css`
- Avoid scattering style logic into many page files unless a component truly needs local logic

4. Respect route and role boundaries
- Many pages are intentionally role-gated or workspace-gated
- Do not widen access casually

5. Be careful with backend assumptions
- Firebase is the target direction
- legacy SQL support still exists
- always inspect the service function being changed

6. For large features, update all layers
- UI
- route permissions
- service function
- cache invalidation
- any server fallback behavior

7. For school-scoped data, keep school isolation intact
- super admin is the platform role
- normal school data must remain school-scoped

GSR ERP is a large, school-first operational platform with:
